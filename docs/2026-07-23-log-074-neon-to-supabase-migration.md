# Log 074 — Neon → Supabase Database Migration & Poll Cadence Update

**Date:** 2026-07-23
**Time:** 06:30 PM (+08:00)

## Summary of Session

Completed the full migration from Neon PostgreSQL to Supabase PostgreSQL, resolving the compute-hour billing issue (Neon's free tier meters CU-hours and our polling defeated its 5-minute scale-to-zero; Supabase's free tier has unlimited compute gated only by a 7-day inactivity pause). Additionally changed the poll cadence from 15 minutes to 5 minutes and updated every doc and config to match.

## Work Completed

### Step 1 — Driver swap (`lib/db/index.ts`, `package.json`)
- Removed `@neondatabase/serverless`. Installed `pg` and `@types/pg`.
- Changed from `drizzle-orm/neon-http` + `neon()` to `drizzle-orm/node-postgres` + `pg.Pool`.
- Pool config: `max: 1`, `idleTimeoutMillis: 5000`, `allowExitOnIdle: true` — the last flag is critical: without it, Next.js static build workers hold the process open indefinitely, causing the build to timeout and fail with "attempt 1 of 3" errors.
- **Deviation from plan:** The original plan specified `postgres-js` (`drizzle-orm/postgres-js`). Switched to `pg` (node-postgres) because `postgres-js` held connections open during `next build` static page generation, causing repeated 60-second worker timeouts regardless of `idle_timeout` settings. `pg` with `allowExitOnIdle` resolves this cleanly.

### Step 2 — Schema migration (`drizzle-kit push`)
- All `.env.local` connection strings swapped to Supabase pooler (port 6543).
- Created `scripts/reset_db.ts` to drop and recreate the public schema, giving a clean slate.
- Ran `drizzle-kit push` against Supabase — all tables created successfully including the `location` column that had previously failed (the column existed in Neon but was missing from a prior Supabase attempt due to schema drift).

### Step 3 — Data migration (`scripts/dump_neon.ts` + `scripts/restore_supabase.ts`)
- **Deviation from plan:** Could not use `pg_dump`/`psql` — raw Postgres CLI tools are blocked in the sandbox environment. Used custom Node.js scripts instead.
- `scripts/dump_neon.ts` — reads all tables from Neon via Drizzle `select()` and serialises to `neon_data.json`.
- `scripts/restore_supabase.ts` — deserialises `neon_data.json` (with ISO date string → `Date` reviver), temporarily changes `GENERATED ALWAYS` identity columns to `GENERATED BY DEFAULT` (to allow explicit ID insertion), inserts all rows with `onConflictDoNothing()`, then restores columns to `GENERATED ALWAYS` and calls `setval()` to advance sequences past the inserted IDs.
- All tables restored successfully: `clans`, `members`, `member_snapshots`, `unit_levels`, `membership_events`, `wars`, `war_attacks`, `war_rosters`, `war_roster_slots`, `capital_district_snapshots`, `capital_raid_seasons`, `hall_of_fame_records`, `cwl_seasons`, `runtime_settings`.

### Step 4 — Bug fixes required post-migration

#### `app/api/cron/purge/route.ts` — `rowCount` vs `count`
The purge route used `prunedSnaps.rowCount` which was correct for the old `neon-http` driver. With the new `pg` driver, `db.execute(sql\`DELETE...\`)` returns a `QueryResult` which also uses `rowCount` — but the TypeScript type narrowing was failing on the intermediate `RowList` type from `postgres-js`. Resolved by switching to `pg` driver which natively exposes `rowCount` on `QueryResult<T>`.

## Decisions Made

- **`pg` over `postgres-js`:** `postgres-js` held connections open during `next build` static page generation (no equivalent of `allowExitOnIdle`), causing worker timeout failures. `pg` pool with `allowExitOnIdle: true` closes idle connections promptly and lets Next.js build workers exit cleanly.
- **Custom dump/restore scripts over `pg_dump`:** Raw Postgres CLI tools are unavailable in the sandbox. The Node.js script approach using Drizzle queries is equivalent in correctness and more auditable (each table is explicitly listed).
- **5 min cadence:** Supabase doesn't meter compute, so the previous concern about Neon CU-hour exhaustion is gone. 5-minute polling gives meaningfully fresher data (war attacks, donation changes, member joins/leaves) with no cost penalty.
