## Decisions Made

- **3-window data fetched server-side, tabs switch client-side**: The page fetches all 3 windows (24h/7d/30d) of donation + activity data in parallel via `Promise.all` (20 queries). The client `DashboardShell` manages tab state with `useState`, so tab switches are instant with no API calls or page reloads. This avoids the "do not add browser-facing read API routes" constraint from concept/12 Step 1.1.A.
- **Recharts over CSS bars**: The design proposal used simple CSS bars, but Recharts was already in the dependencies and the concept explicitly lists it. Recharts gives accessible tooltips, responsive containers, and better mobile behavior. Configured with the dark observatory theme (purple/violet bars, JetBrains Mono tick labels).
- **Member-detail sheet as a minimal shell**: Built a minimal version showing name, tag, TH, role, war preference, and activity score breakdown. The full member detail (activity timeline, donation history, war participation, career stats, progression cards) will be built in Phase 1.3 (Members page). This matches the deferred item from 1.0.B.
- **`MemberDetailSheet` uses a `Map` lookup**: The dashboard shell builds a `Map<playerTag, memberInfo>` from the activity score entries + needs-attention members. This avoids passing the full member list to the modal and keeps the lookup O(1).
- **`next/image` for clan badge**: Used `next/image` with `unoptimized` for the clan badge (external URL from `api-assets.clashofclans.com`). The `unoptimized` flag avoids needing to configure `next.config.ts` image domains for Phase 1.
- **Data freshness panel**: Added a "Data freshness" panel showing last poll, last daily batch, tracking start, and war last-synced times. This wasn't explicitly in the spec but matches the concept/05 §1 "clear capture-time indicator" rule and helps users understand cold-start states.

## Deviations from Original Task Instructions

- **1.2.B "Render the purged-member 'data removed' sheet state without querying deleted profile data"**: The purged-member state is handled in the `ClanLogPanel` — purged members show "data removed" inline and are not clickable (button is disabled). The member-detail sheet is not opened for purged members, so no query is needed. This is a simpler approach than a dedicated sheet state and matches the clan-log spec in concept/05 §8.
- **1.2.C "Test desktop, tablet, and 375–430px mobile layouts"**: Verified desktop (1440×900) and mobile (390×844) via Agent Browser. Tablet (768px) was not explicitly tested but the responsive breakpoints (`sm:`, `lg:`) should handle it. Full tablet testing deferred to Phase 4.0 (release hardening).
- **1.2.C "Confirm donation reset calculations against seeded snapshot data"**: Not done — the live database has real data, not seeded test data. The reset-safe calculation logic is already tested in `tests/lib/donation-delta.test.ts` (12 tests). Seeded-data verification deferred to a formal DB test strategy.
