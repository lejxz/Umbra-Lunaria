## Decisions Made

- **View models as a separate layer**: Chose to define typed interfaces in `lib/view-models/dashboard.ts` rather than returning Drizzle row types directly. This enforces the concept/01 rule "UI components receive display-ready data and never expose API secrets" and makes the UI independent of schema shape changes.
- **Timezone helpers using `Intl.DateTimeFormat`**: Node supports full IANA timezone names via ICU, so `Intl.DateTimeFormat("en-US", { timeZone: "Asia/Manila" })` works without a library like `date-fns-tz` or `luxon`. This avoids an extra dependency.
- **`computeActivityScore` returns `Omit<MemberActivityScore, "rank">`**: The scoring function is pure and doesn't know clan ordering. Rank is assigned by the leaderboard caller after sorting. Pinned with both a runtime test (`not.toHaveProperty("rank")`) and a compile-time test.
- **`inArray()` over raw `ANY()`**: The neon serverless driver doesn't support `ANY(${array})` in parameterized queries (`op ANY/ALL (array) requires array on right side`). Switched to drizzle's `inArray()` helper which generates the correct `IN ($1, $2, ...)` SQL.
- **`getTrackingStart()` string-to-Date conversion**: Raw SQL `min()` returns a string from the neon driver, not a Date. Fixed by typing the column as `sql<string>` and wrapping in `new Date()`.
- **Parallel subagent for scoring + tests**: Delegated `lib/scoring/activity-score.ts` + `lib/scoring/war-metrics.ts` + their tests to a subagent while the main agent built the query layer. The scoring functions are pure and have no DB dependency, making them safe to parallelize.
