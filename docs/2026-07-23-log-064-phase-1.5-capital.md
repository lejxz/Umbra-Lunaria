## Work Completed

### Pure diff logic (`lib/capital/district-diff.ts`)
- `diffDistrictSnapshots()` ‚Äî takes a flat list of (district, level, capturedAt) rows, groups by district, sorts chronologically, and emits an upgrade event for each consecutive pair where the level increased. Level decreases (data corrections) and no-change pairs produce no events. Returns events newest-first. Pure: no DB, testable.

### View models (`lib/view-models/capital.ts`)
- `CapitalPageData`, `CapitalOverview`, `CapitalDistrict`, `DistrictUpgradeEvent`, `DistrictUpgradeHistory`. Every field typed so the UI renders loading/empty/unavailable/pending states without guessing.

### Query layer (`lib/db/capital-queries.ts`)
- `getCapitalPage()` ‚Äî runs `getCapitalOverview()` + `getDistrictUpgradeHistory()` in parallel.
- `getCapitalOverview()` ‚Äî reads the cached clan row (Hall level, points, league, districtsPayload) + the latest district-snapshot capture time.
- `getDistrictUpgradeHistory()` ‚Äî reads all district snapshots, calls the pure `diffDistrictSnapshots()`, computes distinct district names (for the filter), the earliest capture time, and the cold-start flag.

### UI (`components/capital/*`)
- `capital-shell.tsx` ‚Äî client composition root; holds the district-filter state.
- `capital-overview-card.tsx` ‚Äî Hall level / points / districts / league + freshness ("Captured Xm ago") + the "live upgrade cost not provided" unavailable note.

- `upgrade-timeline.tsx` ‚Äî chronological upgrade events (newest-first) with a timeline rail, district filter chips, cold-start state ("Tracking just started"), no-events-for-filter state, and the "tracking began ‚Ä¶ upgrades before this date not recorded" caveat.

- **No new tables.** The Phase 1.0 hardening migration already created `capital_district_snapshots` + the clan capital fields, and the daily batch already captures district snapshots. 1.5 is pure reads + UI ‚Äî no migration, no ingest changes.
- **Diff is pure + tested.** The district-diff logic is extracted into `lib/capital/district-diff.ts` (same mocked-query-boundary pattern as the war/ingest logic) so the "no snapshot / one snapshot / level-increase" test cases run without a DB.
- **Raid pending, not faked.** concept/08 is explicit: don't fabricate a raid leaderboard from lifetime player totals. The pending card says so directly and notes that Phase 3.1 adds the real ingestion.
